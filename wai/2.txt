以下分三个问题逐一解答，均基于实际代码。

---

## 1. 长程任务（大数据量、耗时查询）

WrenAI **没有异步任务队列**，查询是同步阻塞的。它通过以下几个机制来控制长查询：

**① 数据库级别超时（statement timeout）**

通过 HTTP header `x-wren-db-statement_timeout` 传入超时秒数（默认 180 秒），在建立连接时注入到各数据库的会话参数： [1](#1-0) 

| 数据库 | 超时参数 |
|---|---|
| PostgreSQL | `statement_timeout` (session option) |
| ClickHouse | `max_execution_time` |
| Trino | `query_max_execution_time` |
| BigQuery | `job_timeout_ms` |
| MSSQL | `connection.timeout` |

超时触发后，各 connector 捕获 `QueryCanceled` / `TimeoutError` 并向上抛出，不会静默挂起。 [2](#1-1) 

**② LIMIT 下推（pushdown_limit）**

`wren-core-py` 提供 `pushdown_limit` 方法，在 Rust 层通过 AST 操作将 LIMIT 注入 SQL，防止全表扫描： [3](#1-2) 

**③ 工具层硬限制**

LangChain SDK 的 `wren_query` 工具硬限制返回 1000 行；需要更大数据量时需直接调用 `toolkit.query()`： [4](#1-3) 

---

## 2. 查数据的并发设计

WrenAI 的并发设计**相对简单**，没有连接池，主要依赖以下机制：

**① 每次请求独立连接**

各 connector（Postgres、MySQL、ClickHouse 等）在初始化时建立一个连接，查询完成后关闭。没有连接池层。

**② wren-core 的 tokio 异步运行时**

Rust 引擎 `wren-core` 内部使用 tokio 异步运行时，`transform_sql` 是同步包装（`block_on`），Python 侧调用是同步的： [5](#1-4) 

**③ 线程安全的懒初始化**

MySQL connector 的类型映射表使用 `functools.cache` 保证多线程并发初始化时不会暴露半初始化状态： [6](#1-5) 

**④ LangChain SDK 的并发限制**

文档明确说明工具是同步的，LangGraph 会自动桥接到线程池，但**超过 32 个并发用户可能耗尽默认 executor**： [7](#1-6) 

---

## 3. 防止 SQL 注入

WrenAI 有多层防注入机制：

**① MDL 白名单（最核心的防线）**

Agent 只能引用 MDL 中声明的模型对象（Models、Columns、Relationships），物理表名对 agent 不可见。`wren-core` 在 Rust 层通过 DataFusion 的 AST 解析和逻辑计划生成 SQL，**不是字符串拼接**，未声明的表/列在计划阶段直接报错： [8](#1-7) 

**② limit 参数强制类型转换**

MySQL connector 的 `_coerce_limit` 对 limit 值做 `int()` 强制转换，`"1; DROP TABLE foo"` 这类注入字符串会直接抛 `ValueError`： [9](#1-8) 

测试明确覆盖了这个场景： [10](#1-9) 

**③ RLAC session property 值的严格解析**

行级访问控制（Row Level Access Control）的 session property 值通过 `parse_expr` 解析，只允许字面量（数字、字符串、布尔、列表、字典、INTERVAL），以下均被拒绝：

- 函数调用：`upper('aaa')`
- 子查询：`(select 1)`
- 逻辑表达式：`1 or 1`
- CASE 表达式 [11](#1-10) 

**④ ODBC 连接字符串转义**

MSSQL connector 对连接字符串中的所有值用 `_escape_odbc_value` 包裹，防止连接字符串注入： [12](#1-11) 

---

**总结**：WrenAI 的注入防护核心是"结构化 AST 而非字符串拼接"——SQL 经过 `wren-core` 的 Rust 引擎解析成逻辑计划再生成，用户输入的自然语言从未直接拼接进 SQL。长程任务依赖数据库侧超时 + LIMIT 下推，并发能力目前较弱，是其明确承认的限制。

def _coerce_limit(limit: int | None) -> int | None:
    """Validate and coerce a user-supplied ``limit`` to a non-negative ``int``.

    ``int(limit)`` rejects strings like ``"5 OR 1=1"`` so the value can be
    safely interpolated into SQL. Negative limits are also rejected.
    """
    if limit is None:
        return None
    coerced = int(limit)
    if coerced < 0:
        raise ValueError(f"limit must be non-negative, got {coerced}")
    return coerced
