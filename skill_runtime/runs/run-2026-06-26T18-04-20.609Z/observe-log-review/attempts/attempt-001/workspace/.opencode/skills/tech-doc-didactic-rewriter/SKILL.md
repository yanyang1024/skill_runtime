# Tech Doc Didactic Rewriter · 技术文档教学化重写

## 概述

将技术文档、技术方案、架构设计等专业内容，使用 **Cool Teacher（最酷的老师）** 教学法解构并重写为结构清晰、视觉精美的 HTML 教学页面。输出为单个自包含的 HTML 文件，可直接在浏览器中打开阅读。

## 触发条件

用户表达了以下任何意图时，都应考虑使用此 skill：

- 需要将技术方案/文档转化为教学内容
- 需要向非技术人员解释技术概念
- 需要制作技术培训/分享材料
- 说"用大白话讲清楚"、"用类比解释"、"写成教程"
- 说"帮我写一份XX的学习指南/教学文档"
- 说"这个技术文档太晦涩，帮我改成人能看懂的"

## 教学框架：Cool Teacher 五步法

对每一个核心概念，严格按以下五个步骤讲解：

### 1. 概念解构 (Concept Deconstruction)
用一句 **8岁小孩都能听懂的大白话**，说出这个技术术语的本质。彻底告别术语和行话。

> 例："Kubernetes 是一个能帮你自动管理服务器的 '管家'。"

### 2. 神仙比喻 (Analogy Master)
找一个 **日常生活中的场景**（点外卖、寄快递、交通灯、超市购物、做饭……）来精准映射这个技术的工作原理。比喻要贴切，不能牵强。

> 一个好的比喻就是半个老师。读者可能记不住技术细节，但一定记得住好比喻。

### 3. 硬核拆解 (Component Breakdown)
逐步拆解核心组件或步骤。保持语言通俗，适当使用 Emoji 提升可读性。每个步骤配一句功能说明。

格式示例：
```
📦 组件一：XXX（它是做什么的）
├─ 🧩 子组件 A：功能说明
└─ 🧩 子组件 B：功能说明
⚙️ 步骤二：XXX
├─ ① 第一步
├─ ② 第二步
└─ ③ 第三步
```

### 4. Why It Matters (价值与痛点)
用 **对比的方式** 说明：没有这个技术，世界会有什么麻烦？有了它，解决了什么痛点？

> 对比让读者瞬间理解技术的价值。一定要具体，不要空谈"提高效率"。

### 5. 盲点排雷 (Blind Spot Debunking)
指出初学者最容易产生的 **误解**，每个误解配一句简短纠正。

```
❌ 误区：XXX
✅ 真相：XXX
```

## 工作流程

处理每个任务时按以下步骤执行：

### Step 1: 理解输入文档
读取用户提供的技术文档/方案，提取核心概念列表。识别文档的目标读者原设定——但本 skill 的目标读者是"零基础学习者"。

### Step 2: 规划教学结构
将提取的概念按教学逻辑排序（从基础到进阶，从 Why 到 What 到 How）。确定哪些概念需要单独成节，哪些可以合并。

### Step 3: 逐节编写教学内容
对每个核心概念，应用 Cool Teacher 五步法编写内容。注意：
- 保持口语化、亲切感，仿佛在面对面教学
- 不要一次性抛出太多概念，每节聚焦一个核心主题
- 节与节之间要有自然的过渡（"理解了负载均衡，我们来看看另一个重要部件——反向代理"）

### Step 4: 生成 HTML 文件
将编写好的教学内容渲染为单个 HTML 文件（见下方 HTML 输出规范）。

### Step 5: 告知用户
输出文件路径，简要说明教学内容的组织方式，询问是否需要调整。

## HTML 输出规范

输出必须是 **单个自包含的 HTML 文件**，所有 CSS 和 JS 内嵌其中。

### 技术栈
- **样式框架**: Tailwind CSS（通过 CDN `<script src="https://cdn.tailwindcss.com"></script>`）
- **图标库**: Font Awesome 6（通过 CDN）
- **字体**: 中文字体使用系统原生字体栈

### 设计规范

#### 整体布局
- **响应式**：手机端和桌面端都有良好阅读体验
- **留白充足**：段落间距、边距要宽裕，阅读体验舒适
- **内容宽度**：文章内容区 max-w-4xl，居中显示

#### 视觉风格
- **背景**：使用渐变背景（gradient），柔和不宜过于花哨
- **卡片**：每个教学步骤（五步法）使用毛玻璃效果（glassmorphism）卡片
  - `backdrop-filter: blur(16px)`
  - 半透明背景 `rgba(255,255,255,0.7)` 或深色模式对应值
  - 圆角 `rounded-2xl`
  - 微弱的边框和阴影
- **配色**：
  - 主色：蓝色/靛蓝色系（tailwind: blue/indigo），表达专业与信任
  - 强调色：暖色（amber/rose）用于高亮和 Emoji 装饰
  - 整体遵循 light mode，可在页面底部提供暗色模式切换按钮

#### 每个概念节的 HTML 结构

```
┌─ Section Header (概念名称 + 简短引言) ─────────────────────┐
│  渐变背景 + 大标题                                            │
├─ 卡片1: 概念解构 ───────────────────────────────────────────┤
│  Icon + 大白话解释                                            │
├─ 卡片2: 神仙比喻 ───────────────────────────────────────────┤
│  Icon + 比喻场景 + 映射说明                                    │
├─ 卡片3: 硬核拆解 ───────────────────────────────────────────┤
│  Icon + 结构化步骤 (树状/编号)                                 │
├─ 卡片4: Why It Matters ────────────────────────────────────┤
│  Icon + 对比双栏 (没有它 vs 有了它)                             │
├─ 卡片5: 盲点排雷 ──────────────────────────────────────────┤
│  Icon + 误区/真相列表                                         │
└──────────────────────────────────────────────────────────────┘
```

#### 交互细节
- 卡片在进入视口时有轻微的向上滑入动画（fade-in-up）
- 导航锚点平滑滚动
- 可选：页面顶部显示进度条
- 暗色模式切换按钮（使用 `dark mode` 的 Tailwind 支持，或通过 JS + CSS 变量实现）

### 最低可接受标准
即使不做复杂动画，也必须保证：
1. 内容结构清晰（五步法完整呈现）
2. 排版美观（Tailwind 样式一致）
3. 响应式（移动端可用）
4. 配色和谐
5. 教学逻辑完整

## 输出文件命名

默认输出文件名为 `teaching-<概念英文或拼音>.html`，如 `teaching-microservices.html`。如果用户指定了文件名则按用户要求。

## 示例

### 示例1：零基础解释 Kubernetes

**用户输入**："帮我用 Cool Teacher 的方式解释一下 Kubernetes，生成 HTML"

**教学结构参考**：
1. 概念解构：K8s 是一个会自动管理服务器的"管家"
2. 神仙比喻：像一个酒店前台——客人来了安排房间，房间满了引导去别处，有人退房后打扫再分配
3. 硬核拆解：Pod（最小的房间）、Service（前台电话总机）、Deployment（客房部经理的排班表）
4. Why It Matters：没有 K8s → 服务器宕机了只能半夜爬起来手动重启；有 K8s → 自动检测故障重启
5. 盲点排雷：K8s 不是部署代码的工具，而是管理容器的平台

### 示例2：解释技术方案中的微服务架构

**用户输入**："这是我们的微服务架构方案，帮我把它改造成教学文档"

**教学结构参考**：
1. 先整体说明"单体 vs 微服务"的大图景（Why it Matters 前置）
2. 逐一拆解每个服务的职责
3. 服务间通信机制（类比：办公室不同部门之间怎么传递文件）
4. 服务发现与负载均衡（类比：总机接线员）
5. 常见陷阱：分布式事务、最终一致性

## 注意事项

- **不要堆砌技术细节**。教学的目标是让读者理解核心思想，不是成为专家
- **比喻必须贴切**。如果找不到好比喻，宁可不写也比强行用好
- **避免 AI 味的套话**。不要用"总之"、"综上所述"、"值得注意的是"等模板句式
- **教学语气要真实**。像一个有经验的工程师在给同事讲解，而不是一个机器在生成报告
- **每一节都要有画面感**。读者合上眼睛时，应该能"看到"这个技术在工作的画面
